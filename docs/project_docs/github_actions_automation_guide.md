# Руководство по GitHub Actions и автоматизации проекта EasyEPLANner

Этот документ написан для человека, который раньше не работал с GitHub Actions, GitHub Project и автоматизацией репозитория. Ниже описано, что именно делает каждый файл, какие секреты нужны, как устроен процесс, и как собрать такой же проект у себя.

## 1. Что здесь вообще автоматизируется

В репозитории используются GitHub Actions для пяти задач:

1. Автоматическая обработка issue при их создании и изменении.
2. Автоматическое обновление информации об итерациях в шаблонах issue.
3. Автоматическая сборка проекта и проверка кода.
4. Автоматическое обновление версии сборки.
5. Автоматический перенос незавершённых задач на следующую итерацию.

Все эти сценарии живут в папке [.github/workflows](../.github/workflows).

## 2. Что такое GitHub Actions простыми словами

GitHub Actions — это встроенный механизм GitHub, который умеет запускать команды автоматически, когда происходит событие в репозитории.

Примеры событий:

- кто-то открыл issue;
- кто-то изменил issue;
- наступило определённое время по расписанию;
- кто-то запустил workflow вручную;
- кто-то отправил pull request или тег.

Workflow — это файл в формате YAML. В нём описывается:

- когда запускать автоматизацию;
- какие права ей дать;
- какие шаги выполнять;
- какие secrets использовать;
- что делать с результатом.

## 3. Основные файлы автоматизации

### 3.1 [.github/workflows/auto-priority.yml](../.github/workflows/auto-priority.yml)

Этот workflow срабатывает, когда issue создаётся или редактируется.

Что он делает:

- читает приоритет из issue form;
- находит или добавляет issue в GitHub Project;
- ставит нужное значение в поле Priority;
- определяет текущую или следующую итерацию проекта;
- ставит issue в нужную итерацию.

Как он работает по приоритету:

- Low и Long-term plans идут в следующую итерацию;
- Medium, High и Critical идут в текущую итерацию.

Если следующая итерация ещё не создана, workflow может временно поставить задачу в текущую итерацию и оставить комментарий с пояснением.

### 3.2 [.github/workflows/sync-iteration-info.yml](../.github/workflows/sync-iteration-info.yml)

Этот workflow обновляет текстовый блок в issue templates.

Что он делает:

- берёт список итераций из Project;
- определяет текущую итерацию;
- определяет следующую итерацию;
- форматирует текущую дату и диапазоны дат;
- обновляет блок в issue templates, чтобы там всегда была актуальная информация.

Это нужно для того, чтобы пользователь видел актуальные даты прямо в форме issue.

### 3.3 [.github/workflows/auto-move-incomplete-issues.yml](../.github/workflows/auto-move-incomplete-issues.yml)

Этот workflow запускается по расписанию и проверяет незавершённые задачи.

Что он делает:

- берёт итерации из Project;
- смотрит открытые issue, которые уже лежат в итерации;
- если итерация закончилась, переносит issue на следующую;
- оставляет комментарий о том, что задача была перенесена автоматически.

### 3.4 [.github/workflows/autoviersion.yml](../.github/workflows/autoviersion.yml)

Этот workflow обновляет версию сборки.

Что он делает:

- запускается по расписанию раз в месяц;
- обновляет номер версии в [src/Properties/AssemblyInfo.cs](../src/Properties/AssemblyInfo.cs);
- коммитит изменения обратно в репозиторий.

### 3.5 [.github/workflows/msbuild.yml](../.github/workflows/msbuild.yml)

Это основной CI workflow.

Что он делает:

- проверяет, можно ли доверять автору pull request;
- собирает проект на Windows;
- ставит JDK и инструменты для анализа;
- восстанавливает NuGet-пакеты;
- собирает решение;
- запускает тесты;
- отправляет результаты в SonarCloud;
- скачивает дополнительные бинарники и артефакты;
- прикладывает готовую сборку как artifact.

## 4. Issue templates

Issue templates лежат в [.github/ISSUE_TEMPLATE](../.github/ISSUE_TEMPLATE).

Основные файлы:

- [bug_report.yml](../.github/ISSUE_TEMPLATE/bug_report.yml)
- [feature_request.yml](../.github/ISSUE_TEMPLATE/feature_request.yml)
- [exception_report.yml](../.github/ISSUE_TEMPLATE/exception_report.yml)

В каждом шаблоне есть:

- title;
- description;
- dropdown priority;
- textarea для описания проблемы;
- блок с автообновляемой информацией по текущему и следующему спринту.

GitHub Issue Forms не умеет подтягивать данные из Project прямо во время ввода формы. Поэтому информация обновляется workflow и записывается в шаблоны заранее.

## 5. Какие secrets нужны

Автоматизация использует repository secrets. Их нужно создать в настройках репозитория.

### 5.1 Секреты для автоматизации issue и Project

- `AUTOMATISATION_PROJECT` — токен, которым workflow может менять Project и писать комментарии.
- `PROJECT_ID` — ID GitHub Project V2.
- `PRIORITY_FIELD_ID` — ID поля Priority в Project.
- `ITERATION_FIELD_ID` — ID поля Iteration в Project.
- `PRIORITY_LOW_ID` — ID варианта Low в поле Priority.
- `PRIORITY_MEDIUM_ID` — ID варианта Medium в поле Priority.
- `PRIORITY_HIGH_ID` — ID варианта High в поле Priority.
- `PRIORITY_CRITICAL_ID` — ID варианта Critical в поле Priority.
- `PRIORITY_LONG_TERM_PLANS_ID` — ID варианта Long-term plans в поле Priority.

### 5.2 Секреты для CI и версионирования

- `GITHUB_TOKEN` — стандартный токен GitHub Actions, обычно доступен автоматически.
- `SONAR_TOKEN` — токен SonarCloud.
- `EPLAN_API_BIN` — токен доступа к хранилищу EPLAN API binaries.
- `BOT_ACCESS_TO_EPLAN` — токен для коммита автоверсии.

Важно: имена secrets чувствительны к регистру и должны совпадать с тем, что указано в workflow.

## 6. Какие переменные окружения используются

Внутри workflows часто используются значения из secrets как переменные окружения.

Примеры:

- `PROJECT_ID`
- `PRIORITY_FIELD_ID`
- `ITERATION_FIELD_ID`
- `PRIORITY_LOW_ID`
- `PRIORITY_MEDIUM_ID`
- `PRIORITY_HIGH_ID`
- `PRIORITY_CRITICAL_ID`
- `PRIORITY_LONG_TERM_PLANS_ID`

Это делается для того, чтобы не хранить ID прямо в коде workflow.

## 7. Как всё это работает вместе

### Сценарий 1. Пользователь создаёт issue

1. Пользователь открывает шаблон bug report, feature request или exception report.
2. Выбирает приоритет.
3. Видит подсказку с текущей датой и текущим/следующим спринтом.
4. Создаёт issue.
5. Workflow `auto-priority.yml`:
   - находит issue в проекте или добавляет его;
   - ставит priority;
   - ставит iteration по правилу priority -> iteration.

### Сценарий 2. Изменение issue

1. Пользователь редактирует issue.
2. Workflow снова запускается.
3. Если приоритет изменился, проектное поле обновляется.
4. Итерация тоже может пересчитаться.

### Сценарий 3. Автообновление текста в шаблоне

1. Workflow `sync-iteration-info.yml` запускается по расписанию.
2. Он читает Project Iteration напрямую.
3. Берёт текущую дату.
4. Формирует текст с текущим и следующим спринтом.
5. Подменяет служебный блок в шаблонах.

### Сценарий 4. Закрытая задача

1. Пользователь закрывает issue.
2. Workflow может перевести item в Done, если такой workflow включён.
3. При необходимости добавляется комментарий о том, что задача завершена.

### Сценарий 5. Перенос незавершённых задач

1. По расписанию запускается `auto-move-incomplete-issues.yml`.
2. Workflow ищет открытые задачи в закончившейся итерации.
3. Переносит их на следующую итерацию.
4. Оставляет комментарий о переносе.

## 8. Как создать такой же проект у себя

Если вы хотите повторить похожую автоматизацию в своём репозитории, сделайте так:

### Шаг 1. Создайте GitHub Project V2

1. Откройте репозиторий на GitHub.
2. Перейдите в Projects.
3. Создайте Project V2.
4. Добавьте поля:
   - Priority
   - Iteration
   - Status, если он нужен

### Шаг 2. Создайте итерации

1. Откройте настройки поля Iteration.
2. Создайте итерации вручную.
3. Задайте даты начала и длительность, например 2 недели.
4. Убедитесь, что есть текущая и следующая итерация.

### Шаг 3. Создайте secrets

1. Откройте Settings -> Secrets and variables -> Actions.
2. Добавьте все нужные secrets.
3. Проверьте, что значения соответствуют реальным ID полей и вариантов.

### Шаг 4. Добавьте workflows

1. Создайте файлы в [.github/workflows](../.github/workflows).
2. Скопируйте туда логику из этого репозитория.
3. Проверьте права `permissions`.
4. Проверьте cron-расписание.

### Шаг 5. Создайте issue forms

1. Создайте файлы в [.github/ISSUE_TEMPLATE](../.github/ISSUE_TEMPLATE).
2. Добавьте `dropdown` для Priority.
3. Добавьте другие нужные поля.
4. Если нужен блок с текущим спринтом, используйте markdown-блок с маркерами, которые обновляет workflow.

### Шаг 6. Проверьте запуск

1. Создайте test issue.
2. Посмотрите Actions tab.
3. Проверьте, что issue добавилась в Project.
4. Убедитесь, что priority и iteration выставились правильно.

## 9. На что обратить внимание

- GitHub Issue Forms не умеют динамически показывать данные Project в момент заполнения.
- Чтобы информация в шаблонах была актуальной, используется workflow обновления.
- Для следующей итерации важно заранее создавать соответствующие iteration в Project.

## 10. Итог

Эта схема позволяет:

- не вводить данные вручную каждый раз;
- держать формы и Project синхронизированными;
- автоматически подсвечивать текущий и следующий спринт;
- уменьшить количество ошибок при создании задач;
- сделать процесс понятным даже для человека без опыта работы с GitHub.
